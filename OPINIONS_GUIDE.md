# OPINIONS.md 使用指南

本文件說明 AGENTS.md 中「Alfred's Opinions」這段設定的用意，以及如何建立與維護 `~/OPINIONS.md`。
概念參考自 Kun Chen 的文章〈Everyone should have an OPINIONS.md〉。

## 這是什麼

`OPINIONS.md` 不是知識庫，也不是技術筆記。
它是一份精簡、版本控管的文件，記錄你長期、穩定的信念、價值觀與判斷傾向。
用 Chen 的說法，這是「你真正相信什麼」的活地圖，而不是你在各平台上零散發表過的所有意見。

## 為什麼值得建立

1. 自我釐清：把散落在推文、貼文、留言裡的反應，提煉成你真正一貫相信的原則。
2. 代理人對齊：AI agent 在做技術決策、寫文案、評估取捨時，可以讀取這份文件，讓輸出更貼近你的品味與優先順序。
3. 去噪：把玩笑話、一時情緒性的吐槽、針對單一事件的反應，跟真正反映長期信念的判斷分開。

## 應該放什麼

- 對取捨（tradeoff）的一貫判斷，例如「寧可選簡單方案也不要過早抽象化」。
- 對某個領域的長期信念與立場。
- 你重複驗證過、願意公開背書的批評或預測。
- 反映你品味與價值觀的原則。

## 不應該放什麼

- 技術實作細節、程式片段、debug 步驟，這些屬於程式碼或技術文件的範疇，不屬於「信念」。
- 單次的玩笑、吐槽、情緒性反應。
- 只適用於特定情境、無法推廣成一般原則的觀察。

判斷標準很簡單：如果半年後你重讀這句話，仍然點頭同意，它才值得留在 OPINIONS.md。
如果只是當下情緒或特定事件的反應，就不該收錄。

## 建議結構

依領域分類，而不是照時間堆疊。
常見分類可以參考：AI agents、軟體工程、產品、職涯、平台生態、機構與組織。
每一類底下用簡短條列句陳述立場，必要時附上一兩句理由。

隨著內容增加，應該定期重新整理歸類，而不是無限往下疊加新條目。
目標是讓這份文件保持精簡、可讀，並且在概念上前後一致。
如果同一主題出現互相矛盾的條目，代表信念已經改變，應該更新舊條目，而不是並存兩種說法。

## 如何跟 agent 協作流程整合

AGENTS.md 目前的設定是：

> 當你正在處理的工作，會因為參考 Alfred 的觀點而更有幫助時，讀取 ~/OPINIONS.md 來理解。

這代表 agent 不會無條件套用 OPINIONS.md，而是在需要判斷、取捨、表態的情境下，才主動讀取這份文件作為對齊依據。
純技術實作（例如修一個 bug 的具體步驟）通常不需要，但涉及架構取捨、產品方向、對外表達語氣的任務，就適合先讀取。

## 維護方式

### 手動維護（起步階段建議先這樣做）

1. 建立 `~/OPINIONS.md`，先用前述分類搭好骨架。
2. 每次你在對話、文章、程式碼審查中表達出一個明確立場，事後花一分鐘判斷它是不是「半年後仍然同意」的那種信念，是的話就補進對應分類。
3. 每隔一段時間（例如每月）通篇重讀一次，合併重複條目，刪除已經不再相信的舊立場，重新分類讓文件維持易讀。

### 自動化維護（進階，可選）

Chen 的做法是用排程 agent 定期做以下事情：

- 從你的公開寫作來源（例如社群貼文、電子報）抓取新內容，萃取出可能的新信念，並保留原始貼文連結作為佐證來源。
- 比對新萃取出的觀點與既有 OPINIONS.md 內容，偵測「立場漂移」，也就是新表態跟舊條目互相矛盾的情況，並標記出來讓你決定要不要更新。
- 追蹤事實性風險，例如某條意見所依賴的事實前提已經過時，需要重新檢視。
- 用穩定雜湊之類的機制去重，避免同一個警示被重複提醒。

如果要導入自動化，建議先確認來源選取範圍、同步狀態管理方式，以及誰來人工複核 agent 標記出的漂移與過時項目，再逐步放權給排程流程。
